# QA 流水线耗时测试报告

> 问题（两轮相同）：`DLBCL 诊断需要哪些病理检查？`  
> 测试脚本：`scripts/profile_latency.py`（临时 monkey-patch，未改核心问答逻辑）  
> 配置项：`config.yaml` → `max_images`（送入 VLM 的图片硬顶）

## 1. 配置变更

送入 VLM 的图片数量本就可调（`settings.max_images` ← `config.yaml` / 环境变量 `MAX_IMAGES`）。本次将默认从 **4 改为 1**：

```yaml
# config.yaml
max_images: 1   # 送入 VLM 的图片硬顶（选图预算）
```

代码回退默认值同步为 `1`（`backend/app/settings.py`）。选图仍走 Phase1 证据页 → Phase2 seed → Phase3 邻居，但总张数不超过该预算。

## 2. 对比结论（先看这个）

| 指标 | A：`max_images=4`（基线） | B：`max_images=1`（本次） | 变化 |
|---|---:|---:|---|
| 墙钟总耗时 | **67.63 s** | **18.74 s** | **−48.9 s（约 −72%）** |
| `vlm_generate` | **64.35 s**（95.4%） | **17.09 s**（91.8%） | **−47.3 s（约 −73%）** |
| `evidence_gate_qwen` | 2.85 s | 1.20 s | −1.65 s（API 波动，与改图数无直接关系） |
| 送入 VLM 的图 | **4** 张整页 | **1** 张（BCEL-2） | −3 张 |
| 最终展示图 | 1（BCEL-2） | 1（BCEL-2） | 相同 |
| 路由 / seed | hybrid / BCEL-2 | hybrid / BCEL-2 | 相同 |
| 生成模式 | multimodal | multimodal | 相同 |
| 原始 JSON | `latency_profile_20260711-140614.json` | `latency_profile_20260712-164241.json` | — |
| run_id | `20260711-140507-7d40550d` | `20260712-164222-f3a7029c` | — |

**结论：把 `max_images` 从 4 降到 1 后，同题墙钟从约 68s 降到约 19s，几乎全部收益来自 VLM 调用变短（少传 3 张图）。** 瓶颈仍是 `vlm_generate`（仍占 ~92%），但已进入文档预算「≤30s」区间。

展示侧两轮最终都只留 BCEL-2 一张，说明基线多送的 3 张对最终展示无贡献，却显著拉长 VLM 时间。

## 3. VLM 在整条链路中的设计作用

设计原则：**索引便宜通用，看图按需付费**。离线不预渲图；仅 `route ∈ {flowchart, hybrid}` 且选到可渲染页时，才把 PNG 交给 VLM。

### 3.1 流水线位置

```text
normalize → retrieve → evidence_gate(Qwen) → refs / KG
  → gather_figures（预算 = max_images）
  → ★ VLM.generate（有图时替换 Qwen.generate）★
  → format → prune_figures → crop → verify → return
```

### 3.2 三类职责

| 职责 | 设计意图 | 是否阻塞响应 |
|---|---|---|
| **① 看图回答（主）** | 读决策树分支，写带 `[Sn]` 的答案 | **是（同步）** |
| **② 页摘要** | 同一次调用输出 `===PAGE_SUMMARY_JSON===`，回写 BM25 | **是（同一次 HTTP）** |
| **③ bbox 裁剪兜底** | 代码可解析；默认几何优先，提示词已不要求 bbox | 已弱化 |

一句话：**VLM = 流程图答案生成器 + 页摘要采集器**；展示裁剪不是主业。`max_images` 直接放大 ①+② 的输入成本。

## 6. 变化解读

1. **主因明确**：VLM 输入从 4 图 → 1 图，`vlm_generate` 从 64s → 17s，墙钟同比例下降。  
2. **展示质量未因减图变差（本例）**：两轮最终展示页相同（BCEL-2 = seed / workup 页），与「病理检查」意图匹配。  
3. **风险**：Phase1 按 hits 顺序先占预算；若 top hit 不是 seed，`max_images=1` 可能送错页。本轮碰巧 top 临床页即 BCEL-2。若要更稳，可改为「预算优先留给 seed」。  
4. **门控波动**：gate 从 2.8s→1.2s 属模型/网络抖动，不宜算作 `max_images` 收益。  
5. **瓶颈未消失**：单图路径下 VLM 仍占 ~92%；若要感知延迟再降，仍需方案 A（文本先出）或流式。

## 7. 架构与后续建议

当前仍是串行：`… → gather_figures → vlm.generate → …`。

| 优先级 | 动作 |
|---|---|
| 已做 | `max_images: 1`（本报告 B） |
| 可选加固 | 选图时优先保证 seed 占第 1 个名额，再填 hits/邻居 |
| 中收益 | 流式输出；gate 与渲图并行 |
| 高收益架构 | 文本 Qwen 先出答案 + 确定性挂图；VLM 改后台补摘要 |

复现：

```bash
# 确认配置
python -c "from backend.app.settings import load_settings; print(load_settings().max_images)"

# 压测
python scripts/profile_latency.py "DLBCL 诊断需要哪些病理检查？"
```
