## 本周进展（详细）

### 1. 确定技术路线

面向医生群体的 DLBCL 专病问答 Agent，核心链路定为：

**NCCN 决策树语义 + 结构感知切片 → Hierarchical-Filtered-RAG → 校验 Agent 检查输出**

- **语义 + 结构感知切片**：整本 NCCN B 细胞淋巴瘤 PDF 按物理结构解析——临床指南按页保留（含页码标识与跳转链接）、Discussion 按「正文 → 参考文献 → 正文」切段，参考文献单独建模、不经相似度检索，而是通过 `reference_ids` 关联展开。
- **Hierarchical-Filtered-RAG**：分层检索与过滤——先按问题类型路由（流程图 / 证据 / 混合），再叠疾病范围 metadata filter，BM25 召回 + 词面 rerank + 证据门控，流程图问题额外走链接图导航与 VLM 多模态。
- **校验 Agent**：`verifier` 检查 `[Sn]` 引用完整性、问题实体是否被证据覆盖（未覆盖须声明「指南未直接提及」）、是否泄露内部 ID 或自列来源；多模态答案按图片接地放宽误判。

当前已跑通 CLI 原型端到端链路（解析 → 索引 → 问答 → trace 日志），覆盖 workup、一线治疗、复发、预后/突变等典型问题。

### 2. NCCN 通用版：数据处理与检索设计（侧重可扩展性）

#### 2.1 数据处理

| 环节 | 设计 | 可扩展性考量 |
|------|------|-------------|
| PDF 解析 | PyMuPDF 两遍扫描：页类型分类 → footer code 抽取 → 链接反查 → Discussion 物理切段 | 页区间、噪声标记收进 `DocumentProfile` 注册表，新增指南 = 注册 profile，不改解析主流程 |
| 对象模型 | 三类主对象 `GuidelinePage` / `DiscussionChunk` / `ReferenceEntry`，统一扁平为 `SearchDocument` 进索引 | 检索层只认 `SearchDocument`，上层结构变更不影响 BM25/向量接口 |
| 参考文献 | 构建时去重、`entry_id` 全局唯一；运行时 `ReferenceResolver` 直查 JSON，不占 top-k | 任意疾病 article 可复用同一展开逻辑 |
| 链接图 | `classify_edge` 区分 `flow`（决策跳转）与 `navigation`（目录/chrome），写入 `PageLink` | 图导航只跟 `flow` 边，可迁移到其他 NCCN 模块 |

Discussion 切段刻意**不用病名做边界**，改按 NCCN 物理结构（正文页 → 参考文献页 → 下一章正文页）识别 article 边界，病名仅贴标签——整本 PDF 一次解析即可服务 FL、MCL、DLBCL 等多疾病，避免目录页/正文散提病名导致的误切。

#### 2.2 检索算法（Hierarchical-Filtered-RAG）

```text
Query 归一化（中→英实体扩展）
  → 问题类型路由（flowchart / evidence / hybrid）
  → Metadata Filter（page_type + 疾病 scope）
  → BM25 top-k（默认；可选 hybrid：BM25 + 向量 + RRF）
  → LexicalReranker 重排
  → 证据门控（DeepSeek 预筛 / 词面降级）
  → 参考文献关联展开（不占检索名额）
  → EvidenceBundle 组装
  → DeepSeek 文本 / VLM 多模态生成
  → answer_formatter 清洗 → verifier 校验
```

**为可扩展性做的关键取舍：**

1. **索引便宜、看图按需付费**：索引阶段只做 PDF→JSON + BM25 + 链接图，不预渲图、不建向量；命中流程图页才按需渲染页图交 VLM，成本随查询量而非语料规模增长，可扩展到上千篇指南。
2. **检索模式可插拔**：默认 BM25-only（满足 ≤30s 延迟约束）；`EMBEDDING_PROFILES` 注册表 + 独立向量索引目录，hybrid 路径完整保留，切换 `--embedding bge-m3` 即可，无需改代码。
3. **疾病范围注册表**：`DISEASE_SCOPES` 定义 `article_ids` + `module_codes` 过滤规则，环境变量 `TARGET_DISEASE_SCOPE` 切换，检索主流程零改动。
4. **首次命中缓存摘要（PageIndex 懒加载）**：VLM 看图时顺带产出一句可检索摘要写入缓存，下次并入 BM25 语料——缓解流程图页 OCR 文字混乱导致的召回漏，且成本随使用量增长。
5. **前端契约预留**：`QAResult.figures` + `to_web_payload()` 输出稳定 JSON（markdown 正文 + 图片 + 来源 + 校验结果），CLI 稳定后可直出 Web UI / API。

#### 2.3 查询时多模态（流程图专用）

流程图页文字被拍乱、链接关系在纯文本中丢失，因此单独设计「受控链接图导航」：

- 证据门控后选 seed 决策页 → 沿 `flow` 边扩展（同 `module_code`、深度/扇出/预算封顶）→ VLM 看图回答 → 答案驱动裁剪只保留回答引用的图。
- 四道闸防止图片爆炸：边分类、同模块、深度/扇出限制、`MAX_IMAGES` 封顶。

### 3. 对 DLBCL 的适配

在 NCCN 通用底座上，通过配置 + 少量专病逻辑完成 DLBCL 落地：

| 适配点 | 实现 |
|--------|------|
| 检索范围 | `TARGET_DISEASE_SCOPE=dlbcl`：Discussion 过滤 `article_id=dlbcl`，临床指南过滤 `module_code=BCEL` |
| 流程图入口 | `dlbcl_flow_map.py`：意图关键词 → 入口决策页硬编码映射（workup→BCEL-2、一线→BCEL-3、再分期→BCEL-4、复发→BCEL-7、随访→BCEL-9 等），准确优先于通用 BM25 猜测 |
| 决策页识别 | 区分 `BCEL-1..9`（决策流程页）与 `BCEL-A/B/C`（方案表），seed 只选数字决策页 |
| 图导航 | 仅在 `BCEL-*` 模块内跟随 `flow` 边；多链接下游按 query 与 anchor_text 相关性排序取 top fanout |
| 测试 | 补充 `test_graph_navigator` / `test_figure_selection` / `test_qa_figures` 覆盖 seed 选择、预算配额、答案驱动裁剪 |

已用典型问题验证：workup 检查清单、一线治疗路径、R-CHOP 后复发、TP53 预后意义等，输出含 `[Sn]` 引用、流程图 Markdown 图片与关联参考文献。

## 遇到的问题

1. **延迟 vs 语义召回**：本地 CPU 加载 embedding + 编码 query 易超 30s 预算，当前默认 BM25-only，靠中→英 query 扩展补语义；语义改写极强且 BM25 召回明显不足时再切 hybrid，需额外建向量索引。
2. **流程图页文字质量**：PDF 流程图页 OCR/提取文字混乱，BM25 初召回易漏；已通过「首次命中缓存摘要」缓解，但冷启动仍依赖意图 seed 或 rerank 排名。
3. **Discussion 切段边界**：按物理结构切段后大幅改善，但若某章无独立参考文献页、或参考文献与下一章正文同页，仍可能切不准，需结合实际 PDF 人工抽样验证。
4. **VLM 依赖**：未配置 `VLM_API_KEY` 时流程图问题降级为证据摘要模式（不读图），诊疗路径类回答质量明显下降。
5. **交付形态**：当前为 CLI 原型，尚无前端与 HTTP API；医学回答仅供辅助检索，不替代医生判断。
